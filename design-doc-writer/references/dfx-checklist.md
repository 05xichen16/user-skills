# DFX 分析检查清单

撰写 DFX 分析时，逐项审视以下类别，识别与当前设计相关的风险点并给出解决方案。

## 性能（Performance）

• [ ] 高频接口的响应时间是否满足 SLA？

• [ ] 是否存在 N+1 查询或全表扫描？

• [ ] 大数据量场景下是否需要分页、缓存、异步处理？

• [ ] 关联关系是否频繁变化？是否适合动态计算 vs 预存储？

• [ ] 批量操作是否有并发控制和限流？

• [ ] 数据库索引设计是否覆盖核心查询路径？

• [ ] 是否需要读写分离或分库分表？


## 可靠性（Reliability）

• [ ] 服务异常时是否有降级和熔断机制？

• [ ] 关键操作是否有重试和幂等保障？

• [ ] 数据一致性方案（强一致 / 最终一致）是否明确？

• [ ] 是否存在单点故障？

• [ ] 异常场景下数据是否可恢复？

• [ ] 依赖的外部服务不可用时系统行为是否可控？


## 安全（Security）

• [ ] 接口是否有鉴权和权限校验？

• [ ] 用户输入是否做了校验和防注入处理？

• [ ] 敏感数据是否加密存储和传输？

• [ ] 日志中是否避免打印敏感信息？

• [ ] 是否满足最小权限原则？

• [ ] 文件上传/下载是否有大小和类型限制？


## 可维护性（Maintainability）

• [ ] 模块划分是否清晰，职责是否单一？

• [ ] 配置项是否外部化，支持动态调整？

• [ ] 日志是否充分，关键路径是否可追踪？

• [ ] 是否有监控和告警机制？

• [ ] 代码是否遵循 SOLID 原则？

• [ ] 是否易于进行单元测试？


## 可扩展性（Scalability）

• [ ] 新增功能是否只需扩展而无需修改已有代码（开闭原则）？

• [ ] 是否预留了扩展点（策略模式、插件机制等）？

• [ ] 数据模型是否支持未来业务变化？

• [ ] 服务是否支持水平扩展？


## 兼容性（Compatibility）

• [ ] 接口变更是否向后兼容？

• [ ] 数据库变更是否有迁移方案？

• [ ] 新旧版本是否可以共存？

• [ ] 第三方依赖版本是否锁定？


## 可测试性（Testability）

• [ ] 核心逻辑是否可独立测试？

• [ ] 外部依赖是否可 mock？

• [ ] 是否有集成测试覆盖关键路径？

• [ ] 边界条件和异常场景是否有测试用例？


## 使用方式

在设计文档第六章中，从以上清单中筛选出与当前设计相关的条目，以表格形式输出：

| DFX问题 | 问题类别 | 问题描述 | 解决方案 | 备注 |
|---------|---------|---------|---------|------|

每条问题必须包含具体描述和可执行的解决方案，避免空泛表述。
